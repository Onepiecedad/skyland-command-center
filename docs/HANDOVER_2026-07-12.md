## 2. Röstsamtal → Cal.com-bokning → activity-loggning (backend)

**Problem (ursprunglig felrapport):** Ett röstsamtal ledde till en korrekt Cal.com-bokning, men aktiviteten loggades inte i SCC-dashboarden.

**Rotorsak:** `backend/src/routes/voice.ts` tog emot ElevenLabs tool-calls men **ingen branch skrev någonsin till databasen**. Bokningar via `gateway_tool` returnerades till rösten och försvann. Den enda activity-loggningen skedde i `leads.ts` när n8n POST:ade till `/leads/intake` efteråt. Dessutom fanns **ingen Cal.com-integration** i backend alls.

**Åtgärd (commit i skyland-command-center):**
- `backend/src/config.ts` — nya env-vars: `CALCOM_API_KEY`, `CALCOM_EVENT_TYPE_ID`, `CALCOM_API_BASE_URL` (default `https://api.cal.com/v2`). Alla optional.
- `backend/src/routes/voice.ts`:
  - Ny `book_appointment`-tool: bokar via Cal.com v2 API, loggar `voice.booking.created` / `voice.booking.failed` som activity (`event_type: 'booking'`).
  - `gateway_tool` loggar nu varje anrop (booking-relaterade taggas som `booking`-event).
  - Helpers `logVoiceActivity()` (best-effort) och `bookCalcomAppointment()`.
- `backend/src/routes/leads.ts` — `session_uuid` gjord optional med genererat fallback-id så en callback utan det inte tyst droppas (400).

**Bugg fångad under arbetet:** `activities.severity` har en CHECK-constraint (`'info' | 'warn' | 'error'`). Använde först `'warning'` → hade avvisats av Postgres. Rättat till `'warn'`.

**Verifiering:** `tsc --noEmit` grönt.

> **ÅTGÄRD KVAR för Joakim:** Sätt `CALCOM_API_KEY` + `CALCOM_EVENT_TYPE_ID` i `backend/.env`. Lägg till `book_appointment` som custom tool på ElevenLabs-agenten (webhook → `POST /api/v1/voice/tools`). Verifiera Cal.com request-shapen mot ditt konto.

> **Not:** Bokningen loggas som activity, inte kopplad till en kontakt — det kräver `contacts`-tabellen (SCC-22) som inte finns än.

## 3. Telemetri: hemsida → SCC-dashboard (huvudissuen)

**Problem (förtydligat av Joakim):** Hemsida-dashboardens funnel (Engagemang / Lead-händelse / Boknings-klick) visade 0, trots att KPI-korten visade 6 leads och 5 röstsamtal.

**Rotorsak:** KPI-korten och funneln läser från **olika källor**:
- KPI: `prospects` / `voice_calls` (server-side, n8n skriver).
- Funnel: `events`-tabellen (client-side telemetri från hemsidan).

Hemsidans `app/tracker.js` band `voice_start`/`voice_end` till **DOM-klick** (`.orb-wrap`, `.end-btn`) som inte fyrar tillförlitligt när samtalet sker via ElevenLabs-widgeten. Därför var funneln tom. `cta_book_click` matchade bara `calendly`-länkar, inte `cal.com`.

**Åtgärd — tre lager:**

### A) Frontend-telemetri (repo: stitch_skyland_ai_operating_system, DEPLOYAD)
- `app/voice.js` — `voice_start` fyras vid faktisk `startSession`, `voice_end` (med `seconds`) i `onDisconnect`. Riktiga SDK-livscykelhändelser, exakt en gång vardera.
- `app/tracker.js` — tog bort de opålitliga DOM-klick-lyssnarna; vidgade boknings-selektorn till `a[href*="calendly"], a[href*="cal.com"]`.

### B) SCC-backend funnel-robusthet (repo: skyland-command-center, PUSHAD)
- `backend/src/routes/website.ts` — funneln folder nu in prospect-/voice_call-sessioner i `engaged`/`leads` som fallback när client-events saknas. Utökade selecten med `session_uuid`. Sets dedupar → ingen dubbelräkning.
- `frontend/src/pages/WebsiteView.tsx` — etikett "Calendly-klick" → "Bokningsklick".

### C) n8n whitelist-verifiering (INGEN ändring behövd)
- Inspekterade `track-event`-workflowets "Validate & Sanitize"-nod via browser.
- Event-typerna (`voice_start`, `voice_end`, `cta_book_click`) var **redan** whitelistade.
- MEN fält-whitelisten släpper bara igenom `seconds`, inte `duration_seconds`. Därför ändrades `voice.js` att skicka `seconds`. Ingen n8n-ändring behövdes.

**Verifiering (end-to-end, bevisad):**
- POST till produktions-webhooken `https://onepiecedad.app.n8n.cloud/webhook/track-event` med giltigt UUID + de fyra event-typerna → `{"ok":true}` HTTP 200.
- Bekräftade i Supabase att alla fyra rader landade, **inklusive `voice_end` med `seconds: 210` intakt**.
- Bekräftade att typerna räknas av dashboardens funnel-logik.
- Testrader städade bort efteråt.
- Icke-UUID session_uuid → korrekt `{"ok":false}` HTTP 400 (validering fungerar).
